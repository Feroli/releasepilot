## DDD evidence (scene 4)

The challenge is explicit: "Model it with DDD, CQRS and events" and "the domain is the whole
point." Scene 4 maps each DDD tactical pattern to the concrete artifact in the codebase:

| DDD building block | In the code |
| --- | --- |
| Aggregate root | `Promotion` — factory `request(...)`, guards its own invariants, records events, immutable when terminal |
| Value objects | `Environment` (enum with `next()`), `PromotionEligibility` (record) |
| Domain service | `EnvironmentPipeline.requireNextStep(...)` |
| Domain events | `PromotionDomainEvent` recorded on each transition |
| Domain errors | `DomainException` / `DomainErrorCode` — a domain error, never a 500 |
| Ubiquitous language / bounded contexts | `promotion`, `catalog`, `releasenotes` packages |

## Narration (subtitle + voiceover track)

**Title.** ReleasePilot moves application versions through `dev → staging → production`. A
backend built for safe, auditable, reversible promotions.

**The problem.** One version, one step at a time, with no skipping environments. Every move is
approved, tracked, and reversible.

**Promotion is the aggregate.** The core decision is to make `Promotion` the aggregate. It owns
the lifecycle — not the controller, not the database.

**Domain-Driven Design.** The challenge says it plainly: model it with DDD. The aggregate guards
its own rules, and a violation is a domain error, never a 500. Value objects, a domain service,
and domain events complete the model.

**State machine.** A promotion walks a strict state machine: `Requested → Approved → Deploying →
Completed`. Any live step can cancel; approved or deploying can roll back.

**Invariants.** The invariants live inside the aggregate: move exactly one step (aggregate
guard); finish the previous environment first (eligibility snapshot); one active promotion per
app and target (partial unique index); only approvers can approve (`ApproverDirectory`); terminal
states are immutable (`terminal_at` guard).

**Hexagonal architecture.** Adapters talk to ports, so the core never sees HTTP or SQL. Inbound
adapters in, infrastructure out.

**CQRS.** Commands and queries are split. Writes go through the aggregate; reads return
purpose-built DTOs.

**Command flow.** A command loads state and calls the aggregate, then saves the change and the
domain event in one transaction. The API responds `201 Created` right away.

**Async handoff.** After commit, the outbox publisher ships events to RabbitMQ. Consumers run
asynchronously, so audit logging never blocks the response.

**Persistence.** PostgreSQL holds the write model, migrated by Flyway. A partial unique index
enforces one active promotion per app and target under concurrency, backed by an optimistic lock
on `aggregate_version`.

**Ports & adapters.** Deployment, issue tracker, notification, and approver lookup are all ports.
The challenge ships deterministic stubs behind them; the core can't tell. Swap a stub for a real
client and the core never changes.

**Error model.** Domain errors map to precise HTTP status codes — `409` for bad transitions,
`403` for non-approvers — returned as RFC-7807 Problem Details.

**API in practice.** Request a promotion with a simple JSON body, then approve, deploy, and
complete it, and query the full state history.

**Tests.** Tested top to bottom: domain unit tests, API tests with MockMvc, and Testcontainers
integration and end-to-end coverage of Postgres, RabbitMQ, and the async audit.

**Trade-offs.** Deliberate choices: a transactional outbox over direct publish so events are
never lost after commit; CQRS without full event sourcing for separation without the overhead; a
fixed pipeline (configurability is future work); and stub adapters behind ports that stay
replaceable.

**What's next.** Request idempotency keys, materialized read projections, richer deployment
failure modeling, and real integrations with authentication.

**Close.** Run it locally with `docker compose up -d` and `./gradlew bootRun`. ReleasePilot —
thanks for watching.
